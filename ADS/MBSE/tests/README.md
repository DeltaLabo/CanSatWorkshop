# ADS MBSE-wide IVV verification plan

> Modeled definitions added: `ADS-IVV-I-PCPL-ALL` for all-version physical component / physical link inspection, `ADS-IVV-A-CE-ALL` for all-version component exchange presence/consistency analysis, and `ADS-IVV-A-ALLOC-ALL` for all-version functional allocation analysis. No external research was performed; this plan uses saved `tests/references/` material, the ADS v0.2 saved reference corpus, and [`../../../PM&SE/IVV.md`](../../../PM&SE/IVV.md).

This README is the ADS-wide planning entry point. `ADS_MBSE_CORPUS.md` and `model_inventory.json` remain supporting corpus files; this README is the standardized IVV planning index. Detailed test-definition diagrams are created under `ADS/MBSE/tests/<activity-id>/` after plan review/selection.

## 1. SSIV / development-version assumptions

- **Inferred baselines:** ADS `v0.1`, `v0.2`, `v0.3`, and `v1.0` are inferred from folder names and D2 titles; no separate SSIV document was found in the reviewed files.
- **Planning interpretation:** `v0.1` and `v0.2` are incremental development baselines, `v0.3` is a PCB-delivery baseline, and `v1.0` is the integrated CanSat/Backplane/OBCC ADS baseline.
- **Execution rule:** every report shall name the exact target version/SSIV, model baseline, hardware revision, and firmware commit.
- **Current definition scope:** user selected **all versions**; `ADS-IVV-I-PCPL-ALL` covers mandatory PC/PL inspection, `ADS-IVV-A-CE-ALL` covers mandatory CE presence/consistency analysis, and `ADS-IVV-A-ALLOC-ALL` covers mandatory functional allocation analysis for all inferred baselines.
- **Ambiguities:** controlled SSIV/current version, mission/capability/use-case/feared-event IDs, data schemas, environmental levels, truth/reference instruments, and workmanship class need confirmation.
- **Terminology:** `5 Hz` is modeled for v0.1/v0.2 internal ADS collection. v1.0 has OBCC `Pointers`/`Returns`, no ADS Serial0 chain, and no explicit ADS `5 Hz` note.

## 5. Reporting conventions

Store evidence under `ADS/MBSE/tests/results/<activity-id>/` unless a version-specific plan uses `ADS/MBSE/<version>/tests/results/<activity-id>/`. Each report shall identify referenced model views/elements, SSIV/version, as-tested configuration, raw evidence, analysis scripts, deviations, anomalies, pass/fail rationale, waivers, and retest status. Reports should reference the model and this plan rather than duplicate stable model content.

## 6. Coverage gaps and follow-up items

- **SSIV ambiguity:** controlled SSIV/current development version is inferred from folders, not an explicit SSIV artifact.
- **Traceability gap:** no explicit mission, capability, use-case, or feared-event elements are modeled; placeholder IDs need confirmation.
- **v0.3 modeling gap:** PCB delivery has no logical, allocation, CE, or functional-chain views; behavior applies only after integration.
- **v1.0 modeling gap:** no ADS Serial0 chain and no explicit ADS 5 Hz constraint are modeled.
- **v0.2 fault-hardening gap:** no explicit I2C read timeout constraint is modeled, although v1.0 includes one.
- **Definition gap:** for remaining Candidate activities, verification means, fixtures, data schemas, and pass/fail constraints are not yet modeled as detailed verification definitions.
- **Execution assumptions:** GPS truth source, angular-rate reference, environmental levels, PCB workmanship class, confidence targets, and firmware commits must be fixed before execution.
- **Definition-stage handoff:** `ADS-IVV-I-PCPL-ALL`, `ADS-IVV-A-CE-ALL`, and `ADS-IVV-A-ALLOC-ALL` have been converted into modeled verification definitions under `ADS/MBSE/tests/` without editing source ADS diagrams. Remaining Candidate activities still need modeled definitions before execution.
