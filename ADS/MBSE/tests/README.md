# ADS MBSE-wide IVV verification plan

> Modeled definitions now cover ADS-wide PC/PL inspection, component-exchange analysis, allocation analysis, GPS/angular/attitude/init/Serial0/OBCC/mission-window functional activities, and GPS/rate/gyro/no-blocking/getter/timing/timeout/manufacturing constraints. Detailed activity IDs and statuses are listed in §4. No external research was performed; this plan uses saved `tests/references/` material, the ADS v0.2 saved reference corpus, and [`../../../PM&SE/IVV.md`](../../../PM&SE/IVV.md).

This README is the ADS-wide planning entry point. `ADS_MBSE_CORPUS.md` and `model_inventory.json` remain supporting corpus files; this README is the standardized IVV planning index. Detailed test-definition diagrams are controlled under `ADS/MBSE/tests/<activity-id>/` for the model-defined activities listed below.

## 1. SSIV / development-version assumptions

- **Inferred baselines:** ADS `v0.1`, `v0.2`, `v0.3`, and `v1.0` are inferred from folder names and D2 titles; no separate SSIV document was found in the reviewed files.
- **Planning interpretation:** `v0.1` and `v0.2` are incremental development baselines, `v0.3` is a PCB-delivery baseline, and `v1.0` is the integrated CanSat/Backplane/OBCC ADS baseline.
- **Execution rule:** every report shall name the exact target version/SSIV, model baseline, hardware revision, and firmware commit.
- **Current definition scope:** user selected **all versions**. Existing all-version definitions cover mandatory PC/PL inspection (`ADS-IVV-I-PCPL-ALL`), CE presence/consistency analysis (`ADS-IVV-A-CE-ALL`), allocation analysis (`ADS-IVV-A-ALLOC-ALL`), and primary GPS/IMU/attitude chains (`ADS-IVV-FC-GPS`, `ADS-IVV-FC-ANG`, `ADS-IVV-FC-ATT`). Additional modeled definitions now cover initialization, Serial0 logging, OBCC delivery, mission-window runtime, position/rate/gyro constraints, no-blocking, Variable Getter, process timing, UART/I2C timeouts, and manufacturing/workmanship. All remain execution pending until reports bind evidence to a selected SSIV/version and as-tested configuration.
- **Execution binding:** the exact SSIV/current version, data-schema revision, environment record, truth/reference instruments, workmanship class, article IDs, firmware commits, and calibration IDs are report-time configuration records. The pass/fail gates and required evidence fields are controlled by the activity definitions below and are not left open by those execution-record values.
- **Terminology:** `5 Hz` remains source-modeled for v0.1/v0.2 internal ADS collection. For v1.0, ADS-to-OBCC freshness planning is controlled by the shared PM&SE contract [`PM&SE/contracts/sensor_obcc_freshness_contract.md`](../../../PM&SE/contracts/sensor_obcc_freshness_contract.md).
- **ADS v1.0 freshness alignment path:** `ADS-V10-DATA-FRESHNESS` is the defined alias implemented by explicit extensions of `ADS-IVV-FC-OBCC-DELIVERY`, `ADS-IVV-C-GETTER`, and `ADS-IVV-C-RATE-5HZ`; internal `5 Hz` freshness evidence is separate from `2 s` LoRa telemetry packaging.
- **Shared freshness oracle:** fresh ADS data requires `status == VALID` and `age_ms <= 400 ms`; the exact enum is `VALID`, `STALE`, `NO_DATA`, `TIMEOUT`, `SENSOR_FAULT`, `INIT_FAIL`; stale/no-data/timeout/fault/init-fail conditions shall not leave old ADS data marked `VALID`. Affected D2/model views and Markdown definitions now identify the v1.0 freshness path; execution remains pending.
- **Accuracy definitions:** `ADS-IVV-C-ACCEL-3AXIS` and `ADS-IVV-C-HEADING-NORTH` are now model-defined verification activities for acceleration units/accuracy and heading/north accuracy, with D2 definition views in their activity folders.
- **Accuracy truth-method decisions:** strict GPS `<5 m` evidence uses GNSS simulator or surveyed/open-sky reference truth; phones are auxiliary/limited unless independently uncertainty-controlled. Strict gyro `<30 deg/s` evidence may use motors plus calibrated tachometer with `n = 59` all-within-limit samples per axis/rate condition.

## 5. Reporting conventions

Store evidence under `ADS/MBSE/tests/results/<activity-id>/` unless a version-specific plan uses `ADS/MBSE/<version>/tests/results/<activity-id>/`. Each report shall identify referenced model views/elements, SSIV/version, as-tested configuration, raw evidence, analysis scripts, deviations, anomalies, pass/fail rationale, waivers, and retest status. Reports should reference the model and this plan rather than duplicate stable model content.

## 6. Coverage gaps and follow-up items

- **SSIV ambiguity:** controlled SSIV/current development version is inferred from folders, not an explicit SSIV artifact.
- **Traceability note:** no explicit mission, capability, use-case, or feared-event elements are modeled as separate ADS source nodes; the provisional IDs above remain controlled trace labels until such nodes are added. No pass/fail gate depends on unresolved trace-label naming.
- **v0.3 modeling gap:** PCB delivery has no logical, allocation, CE, or functional-chain views; behavior applies only after integration.
- **v1.0 execution follow-up:** no ADS Serial0 chain is modeled; v1.0 internal freshness is defined through `ADS-V10-DATA-FRESHNESS` and must be executed without using the `2 s` LoRa cadence as internal-rate evidence.
- **v0.2 fault-hardening gap:** no explicit I2C read timeout constraint is modeled, although v1.0 includes one.
- **Accuracy execution follow-up:** `ADS-IVV-C-ACCEL-3AXIS` and `ADS-IVV-C-HEADING-NORTH` are model-defined and execution/report evidence remains pending before claiming acceleration or heading closure.
- **GPS/gyro truth-method status:** active Markdown criteria select GNSS simulator or surveyed/open-sky GPS truth with phones auxiliary/limited, and motor/tachometer or calibrated rate-table gyro truth with strict `n = 59` evidence where formal gyro claims are made; execution reports record exact assets/calibration.
- **Definition status:** active ADS activities in §4 are model-defined. Remaining limitations are controlled N/A or execution-specific evidence cases: v0.3 runtime behavior, v1.0 ADS Serial0, v0.2 I2C timeout instrumentation path, and execution-specific schemas/tolerances/equipment.
- **Execution records:** surveyed GPS coordinates or simulator scenario, motor/tachometer asset/calibration and axis/rate matrix, Serial0 schema/parser version, ADS/OBCC implementation field-name mapping to the shared freshness evidence fields, timing instrumentation hooks/overhead, PCB workmanship class, hardware revisions, firmware commits, equipment IDs/calibration, environmental readings, and selected report confidence claim must be recorded before execution. Definition-level gates already select the 10 min/600 s mission window, 3000 expected 5 Hz opportunities, 400 ms max progress/freshness gap, exact status enum, and `n = 59` formal timing/freshness observations.
- **Definition-stage handoff:** all §4 verification activities are now represented by modeled definitions under `ADS/MBSE/tests/` without editing source ADS diagrams. Reports should reference the relevant activity README and definition views, then record execution evidence under `ADS/MBSE/tests/results/<activity-id>/<version>/`.
